# Project: League of Legends "Win Contribution" Analysis (SHAP)

## 1. Project Overview & Objective

We are building a Python ML pipeline to quantify the **win contribution** of each of the 10 players in a League of Legends match.

- **Goal:** Train a model on minute-by-minute game state data to predict win probability at every minute of a game.
- **Analysis:** Use **SHAP** to attribute the model's output to individual player features. Sum a player's SHAP values across all their features to get their total contribution to the win/loss.
- **Output:** A per-player contribution score (and eventually a tier rating S/A/B/C) for any given match.

**Why high accuracy matters:** SHAP attributions are only meaningful if the underlying model is genuinely predictive. A low-accuracy model produces noisy, unreliable SHAP values.

List of all champion masteries for a player. Key fields: `championId`, `championPoints`, `championLevel`, `lastPlayTime`.
**Important:** No win rate or games-played available. `championPoints` is the best proxy for champion experience.
**Important:** This is *current* mastery data, not historical — slight staleness vs. game timestamp.

### Event Data (inside `frames[T].events`)

## 5. Modelling Plan — TWO MODELS

The decision was made to build two complementary models:

### Model A — Snapshot Model (`src/04a_train_snapshot.py`) — BUILD FIRST
**Concept:** Treat each `(game, minute)` row independently. Predict `blue_win` from the game state at a single minute T.

- **Model:** LightGBM (fast CPU training, handles categoricals natively, excellent SHAP support)
- **Input:** Single feature vector (~400 features) at minute T
- **CV:** `GroupKFold(n_splits=5)` grouped by `game_id` — critical to prevent same-game leakage across train/val
- **Categorical features:** Pass `*_champion_id`, `*_summoner*_id`, `*_keystone`, `*_primary_tree` as LightGBM categoricals (integers, no encoding needed)
- **Key hyperparams (starting point):** `num_leaves=127`, `learning_rate=0.05`, `n_estimators=1000`, early stopping on val AUC
- **Evaluation:** AUC per minute bucket (0-5, 5-10, 10-15, 15-20, 20-25, 25+), calibration curve, feature importance
- **Expected AUC:** ~0.62 at min 5, ~0.80 at min 15, ~0.94 at min 30+

**SHAP attribution (later step):** Sum SHAP values across all features belonging to a player (by slot name prefix, e.g., all `blue_top_*` features) to get that player's contribution to the win probability.

### Model B — Time-Series Model (`src/04b_train_timeseries.py`) — BUILD SECOND
**Concept:** Feed the full sequence of per-minute feature vectors from minute 1 to T. The model sees history — giving it context that a single snapshot lacks (e.g., a rising gold lead means more than a static one).

- **Model:** PyTorch LSTM (start here), then optionally Transformer
- **Input:** Sequence of feature vectors shape `(seq_len, num_features)` per game, **padded** to max game length with a mask
- **Training split:** 80/10/10 train/val/test at the **game level** (not row level!)
- **Loss:** Binary cross-entropy averaged over non-padded timesteps
- **Architecture:** Linear projection → 2-layer LSTM (hidden=256) → Linear → Sigmoid per timestep
- **CPU-aware settings (NO GPU):** Smaller model, fewer epochs. Recommend:
  - `hidden_size=128` (not 256) for faster CPU training
  - `num_layers=2`
  - `batch_size=32` games (not 64)
  - `max_epochs=30` with early stopping (patience=5)
  - Use `torch.set_num_threads(N)` to use all CPU cores
- **Evaluation:** AUC at each minute T — compare directly to Model A

**Transformer stretch goal:** Same concept but with causal attention mask (position T cannot attend to T+1). Use `d_model=64`, `nhead=4`, `num_layers=3`. More powerful but slower on CPU.

## 7. Design Decisions & Rationale

1. **Event data excluded for now.** Simple aggregation (kills per minute, etc.) adds marginal value over the timeline snapshot features. A time-series model that ingests raw event sequences would be far more powerful, but complex. Deferred to a later phase.

2. **LightGBM over XGBoost.** Faster training, better handling of categoricals natively, comparable accuracy.

3. **Leave-game-out CV.** All minutes of a game must go to the same fold. Using random row splits would cause severe leakage (minute 14 of a game in train, minute 15 in val — the model just memorises game state).

4. **CPU-only training.** No GPU available. LightGBM trains well on CPU. PyTorch LSTM should use `torch.set_num_threads()` to exploit all cores. Keep model sizes conservative (hidden=128 for LSTM, not 256+).

5. **Parquet + Snappy compression.** 825k rows at 420 MB on disk. Load into memory at ~1.3 GB as float32. Fully manageable.

6. **SHAP attribution approach.** For a player with slot `blue_top`, their contribution = sum of SHAP values for all columns prefixed with `blue_top_`. Do this at every minute to get a contribution timeline, then aggregate (mean or final-minute value) for the match summary score.

7. **Mastery data is current, not historical.** `days_since_last_played` is computed as `(gameCreation - lastPlayTime)` but the mastery file was fetched recently. For Challenger players who play frequently, this is still a good recency proxy. Known limitation.

8. **Surrender games included.** 34% of games end in surrender. These are valid competitive games. The model should learn that surrenders tend to follow large gold/objective deficits. No filtering needed.
